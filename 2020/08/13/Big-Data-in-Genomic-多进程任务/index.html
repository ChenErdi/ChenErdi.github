Big Data in Genomic 多进程任务

发表于 2020-08-13 分类于 Python ， Multiprocessing 阅读次数：

Seminararbeit：多进程任务介绍

项目名为Big Data in Genomic，对于数十个TB数据量的基因数据，串行运行将付出高昂的时间代价，
任务是对已有的串行方案进行多进程优化，通过多核同时运行，减少代码运行时间，提高效率。

定义并行方案

采用生产者-消费者模式，分别构建三类工人：worker reader，worker encoder，worker writer。
两个任务队列：queue_reader 与 queue_encoder
其中，reader负责读取源数据，通过readline方法，每读取一个blocksize大小的数据就将其放入一个块（block）中，放入队列queue_reader，每个block包含：block_ID与data。
encoder 负责获取队列queue_reader上的数据，将其编码处理后，所得结果放入队列queue_writer。
writer 负责获取队列queue_writer上的数据，以block_ID顺序写入文件中。

在该方案中，总共使用了n个进程处理，其中，一个进程运行reader，n-2个进程运行encoder，1个进程运行writer。
实际运行中，n设置为了10和16，(根据实际硬件情况设置进程数量)。

代码实现

Reader 部分

由于基因数据往往几个G，甚至几个T，将其一次性读入内存显然是不明智的，通过readline方法每次只读取一行数据，可以避免内存过载。
使用lock，可以有效避免进程冲突。
buffer_size提供一个缓存区，reader只有当队列中的block数据被encoder取走后才放入下一个block。
当所有数据被读取完，即EOF(End of lines)时，发出Stop信号。

Encoder部分

由于多个Encoder与Reader会同时使用队列queue_reader，因此需要使用lock避免进程冲突。
两个while True循环，是为了确保每个Encoder在不断地试图获取下一个block，直到获得”Stop”信号。
只要Encoder获得block，便进行编码处理，不再“争抢”其他的block，直到当前的block处理完毕，然后将结果放入队列queue_encoder。
一旦Encoder接收到”Stop”信号，它将同样地发送”Stop”信号告知Writer。

Writer部分

这里同样使用了两个While True循环，目的是确保Writer获取block，并按顺序0~n写入文件。
同样，考虑到多个Encoder与Writer会同时使用队列queue_encoder,这里也使用了lock，避免进程冲突。
这里的字典也是起到了缓存区的作用，只有当目前的字典中没有需要写入的数据，且不再得到下一个block，循环停止。

启动进程部分

总共使用了n个进程处理，其中，一个进程运行reader，n-2个进程运行encoder，1个进程运行writer。
这样的考虑是基于该项目基因大数据本身的实际情况，数据量大，主要的瓶颈在于encoder编码部分，因此使用该方案。

  1. 低级错误：在Reader中，一开始使用的是

    这导致程序一直报错: Index out of range. 原因是在该循环中，lines被不停的更新，而所有的lines却指向同一内存地址，导致只有第一个block里有数据，而其余的block里都为空。

  2. 一开始使用了queue.empty()作为循环的判断条件，由于queue.empty()是不可靠的，在不使用lock.acquire()的情况下会得到不可靠结果，进而引起进程等待。改为使用queue.qsize()，并使用lock确保qsize可靠。

  3. 代码中大量使用了while True循环，跳出循环的逻辑需要审慎考虑，因为一旦出现纰漏将会使得程序无限运行下去。历经数次更改，最终得到了切实可行有效的解决方案。
